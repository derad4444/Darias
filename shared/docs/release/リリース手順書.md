# リリース手順書

> DARIASアプリのバージョン更新・リリース手順。Web版・iOS版それぞれの作業をまとめた手順書。

**最終更新日**: 2026-05-24

---

## バージョン番号の仕様

```
version: X.Y.Z+N
         ↑ ↑ ↑  ↑
         │ │ │  └── ビルド番号（+N）: リリースのたびに +1
         │ │ └───── パッチ（バグ修正のみ）
         │ └─────── マイナー（新機能追加）
         └───────── メジャー（大規模変更）
```

| バージョン例 | 使いどころ |
|------------|----------|
| `2.2.1+7` | バグ修正のみ |
| `2.3.0+7` | 新機能追加 |
| `3.0.0+7` | 大規模リニューアル |

---

## 変更が必要なファイル一覧

| ファイル | 変更箇所 | 備考 |
|--------|---------|------|
| `flutter/pubspec.yaml` | `version:` の値 | **必須・最初に変更** |
| `flutter/ios/Runner.xcodeproj/project.pbxproj` | `MARKETING_VERSION` (3箇所) と `CURRENT_PROJECT_VERSION` (3箇所) | Profile / Debug / Release の Runner 分のみ（RunnerTests の 1.0 / 1 は変更しない） |
| Android は `pubspec.yaml` の変更のみ | `versionCode` / `versionName` は pubspec.yaml から自動生成 | `project.pbxproj` の変更は不要 |

> `flutter/ios/Flutter/Generated.xcconfig` は `flutter build ios --release` を実行すると **自動更新** されるため手動で変更しない。

## Web版リリース手順

### ① バージョン更新

```yaml
# flutter/pubspec.yaml
version: 2.3.0+7  # ← ここだけ変更
```

### ② ビルド

```bash
cd flutter
flutter build web --release
```

### ③ Firebase Hosting にデプロイ

```bash
# プロジェクトルート（DARIAS/）で実行
firebase deploy --only hosting:darias-app --project my-character-app
```

### ④ コミット＆プッシュ

```bash
git add flutter/pubspec.yaml
git commit -m "chore: バージョンをX.Y.Z+Nに更新"

```

## iOS版リリース手順

### ① バージョン更新（2箇所）

**1. `flutter/pubspec.yaml`**
```yaml
version: 2.3.0+7  # バージョン名+ビルド番号
```

**2. `flutter/ios/Runner.xcodeproj/project.pbxproj`**

以下の3箇所を変更（Runner の Profile / Debug / Release）:
```
MARKETING_VERSION = 2.3.0;      # X.Y.Z 部分（+N は含めない）
CURRENT_PROJECT_VERSION = 7;    # +N の数字
```

> ⚠️ `RunnerTests` の `MARKETING_VERSION = 1.0;` / `CURRENT_PROJECT_VERSION = 1;` は**変更しない**

### ② IPA ビルド

```bash
cd flutter
flutter build ipa --release
```

完了すると `build/ios/ipa/darias.ipa` が生成される（所要時間：数分）。

> ⚠️ Xcode の `Product → Archive → Distribute App` は使わない。
> 一部のパッケージ（`objective_c.framework` 等）にシミュレーター用 x86_64 スライスが混入し、
> App Store Connect へのアップロード時にエラーになるため。

### ③ App Store Connect にアップロード

**Mac App Store の「Transporter」アプリを使う（推奨）：**

1. Transporter を起動（未インストールの場合は Mac App Store からインストール）
2. `build/ios/ipa/darias.ipa` を Transporter にドラッグ＆ドロップ
3. 「配信」ボタンをクリック

**コマンドラインの場合：**

```bash
xcrun altool --upload-app --type ios \
  -f build/ios/ipa/darias.ipa \
  --apiKey <your_api_key> \
  --apiIssuer <your_issuer_id>
```

### ⑤ コミット＆プッシュ

```bash
git add flutter/pubspec.yaml flutter/ios/Runner.xcodeproj/project.pbxproj
git commit -m "chore: バージョンをX.Y.Z+Nに更新"
git push origin main
```

## Android版リリース手順

### 事前準備（初回のみ・設定済みならスキップ）

- `flutter/android/key.properties` — 署名キー情報（gitに含めない）
- `~/darias-release.keystore` — 署名キーストア（絶対に紛失しないこと）
- `flutter/android/app/build.gradle.kts` — 署名設定（`signingConfigs.release` 参照済み）

### ① バージョン更新

`flutter/pubspec.yaml` の `version:` のみ変更（Android は pbxproj 変更不要）。

### ② AAB ビルド

```bash
cd flutter
flutter build appbundle --release
```

生成物: `flutter/build/app/outputs/bundle/release/app-release.aab`（所要時間：数分）

### ③ Google Play Console にアップロード

1. **[製品版リリースを作成（直リンク）](https://play.google.com/console/developers/app/com.darias.darias/tracks/production/create-release)** を開く
2. 「新しいリリースを作成」→ AABファイルをアップロード
3. リリースノートを入力して審査提出

> ⚠️ `versionCode`（+N の数字）は毎回 **必ず +1** すること。同じ versionCode は再アップロード不可。

### ④ コミット＆プッシュ

```bash
git add flutter/pubspec.yaml flutter/ios/Runner.xcodeproj/project.pbxproj
git commit -m "chore: バージョンをX.Y.Z+Nに更新"
git push origin main
```

## Web + iOS + Android 同時リリース手順（まとめ）

```
1. pubspec.yaml の version を更新
2. project.pbxproj の MARKETING_VERSION / CURRENT_PROJECT_VERSION を更新（3箇所）
3. flutter build web --release
4. firebase deploy --only hosting:darias-app --project my-character-app
5. flutter build ipa --release
6. build/ios/ipa/darias.ipa を Transporter でアップロード
7. flutter build appbundle --release
8. build/app/outputs/bundle/release/app-release.aab を Google Play Console にアップロード
9. git add / commit / push
```

## よくあるミス

| 症状 | 原因 | 対処 |
|-----|------|------|
| App Store Connect アップロード時に「Invalid architecture / x86_64」エラーが出る（Xcode Archive経由） | Xcode Archive 経由だと一部パッケージにシミュレーター用スライスが混入する | `flutter build ipa --release` で IPA を生成し Transporter でアップロードする |
| App Store Connect アップロード時に「Invalid executable. objective_c … x86_64 slice」エラーが出る（`flutter build ipa` でも発生） | `objective_c` パッケージの hooks_runner が x86_64 版 dylib をアーカイブに混入させる既知の問題 | 下記「objective_c x86_64 スライス修正手順」を参照 |
| Xcode が「file modified by another application」ダイアログを出す | `project.pbxproj` を外部ツールで編集した | **「Use Version on Disk」** を選択 |
| Web版のバージョンが反映されない | `flutter build web --release` を再実行せずにデプロイした | ビルドしてからデプロイ |

## objective_c x86_64 スライス修正手順

`flutter build ipa --release` 後に Transporter で「Invalid executable … x86_64 slice」が出た場合の対処。

```bash
ARCHIVE="flutter/build/ios/archive/Runner.xcarchive"
FRAMEWORK="$ARCHIVE/Products/Applications/Runner.app/Frameworks/objective_c.framework"
SIGN_ID="Apple Distribution: ryosuke onodera (UYP6X4YKVM)"
IPA_DIR="flutter/build/ios/ipa"

# arm64版 dylib（0b28f89409/ 配下）に差し替え
cp "flutter/.dart_tool/hooks_runner/shared/objective_c/build/0b28f89409/objective_c.dylib" \
   "$FRAMEWORK/objective_c"

# フレームワーク再署名
codesign --force --sign "$SIGN_ID" "$FRAMEWORK"

# IPA を元のパスに再エクスポート（上書き）
xcodebuild -exportArchive \
  -archivePath "$ARCHIVE" \
  -exportOptionsPlist "$IPA_DIR/ExportOptions.plist" \
  -exportPath "$IPA_DIR"
```

`flutter/build/ios/ipa/darias.ipa` を Transporter でアップロードする（ファイルは上書きされる）。
