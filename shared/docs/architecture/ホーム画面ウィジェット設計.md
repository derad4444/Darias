## 1. 概要

DARIASアプリのホーム画面ウィジェット機能。FlutterアプリがFirestoreから取得したデータを共有ストレージ（iOS: App Group UserDefaults、Android: SharedPreferences）にキャッシュし、ネイティブ実装のウィジェットがそれを読み取って表示する。

**設計方針:**
- Flutterアプリが**データ書き込み**を担当
- ネイティブウィジェットが**データ読み取り・表示**を担当
- Flutterとウィジェットはリアルタイム通信せず、**共有ストレージ経由**でデータ連携
- ウィジェットタップで対応アプリ画面にディープリンク遷移

## 4. データフロー

### 書き込みフロー（Flutter → 共有ストレージ）

```
1. アプリ起動 / フォアグラウンド復帰
   └→ MainShellScreen が各データの監視を開始

2. Firestore リアルタイム更新受信
   ├→ allSchedulesProvider (StreamProvider) → CalendarController
   │   ├→ schedules 更新時: WidgetDataService.cacheSchedules() 呼び出し
   │   └→ tagsProvider 更新時: 再キャッシュ (タグ色レースコンディション対策)
   ├→ memosProvider → MainShellScreen
   │   └→ memos 更新時: WidgetDataService.cacheMemos() 呼び出し
   └→ todosProvider → MainShellScreen
       └→ todos 更新時: WidgetDataService.cacheTodos() 呼び出し

3. WidgetDataService
   ├→ JSON エンコード
   ├→ HomeWidget.saveWidgetData() で共有ストレージに書き込み
   └→ HomeWidget.updateWidget() でウィジェット更新通知

4. ネイティブウィジェット
   └→ 通知受信 → WidgetDataCache.load~() でデータ読み取り → UI更新
```

### 読み取りフロー（共有ストレージ → ウィジェット）

```
iOS:
  WidgetDataCache.shared.loadSchedules()
    └→ UserDefaults(suiteName: "group.com.derad.Character")
    └→ キー: "widget_schedules_cache"
    └→ JSON文字列 → [WidgetSchedule] デコード

Android:
  SharedPreferences 経由 (home_widget パッケージが管理)
    └→ キー: "widget_schedules_cache"
    └→ JSON文字列 → Kotlin で手動パース
```

| Provider | 更新間隔 | 備考 |
|---------|---------|------|
| `CalendarWidgetProvider` | 15分 | 予定リスト用 |

**placeholderとgetSnapshot の違い:**
- `placeholder()`: Xcode プレビュー・ウィジェットギャラリー用のダミーデータ
- `getSnapshot()`: ウィジェット追加時のプレビュー。`isPreview` の場合はダミー、それ以外は実データ
- `getTimeline()`: 実際のウィジェット表示に使用

### WidgetData.swift の設計

日付を `Date` 型ではなく **ISO8601 文字列 (String)** で保持する設計。

**理由**: `Date` 型をそのまま UserDefaults 経由で受け渡すと、タイムゾーン情報の不整合でデコードエラーが起きるため。

**日付パース方法** (`startDateParsed` プロパティ):
```swift
// 1. タイムゾーン情報あり ("Z"や"+09:00") → ISO8601DateFormatter でパース
// 2. タイムゾーン情報なし (ローカル時刻) → DateFormatter でJSTとして解釈
//    ※ "Z"を付けてUTCとして解釈すると+9h ずれるため使わない
let fmt = DateFormatter()
fmt.dateFormat = "yyyy-MM-dd'T'HH:mm:ss.SSS"
```

### カレンダーグリッドウィジェット（LargeCalendarGridView）の設計

**行高さの分配方式:**
- `frame(maxHeight: .infinity)` を各行に適用することで、利用可能なスペースを行数で均等分配
- 5週の月 → 1行あたり広め、6週の月 → 1行あたり狭め（Androidの `layout_weight="1"` 相当）
- セルがオーバーフローした場合は `.clipped()` で切り取り

**1セルあたりの表示内容:**
- 日付数字（固定サイズ: 20pt円）
- 祝日名（赤背景、font 6.5pt）
- スケジュール最大3件（font 6.5pt）
  - 終日予定: タグ色背景・白文字
  - 時間指定予定: タグ色テキスト
  - 4件以上: `+N` のオーバーフロー表示

**日付数字の色分け:**
| 曜日 | 色 |
|-----|-----|
| 日曜・祝日 | 赤 (red.opacity(0.8)) |
| 土曜 | 青 (blue.opacity(0.8)) |
| 平日 | テキストプライマリ (#2D2D2D) |
| 今日 | 白 (ピンク円背景) |

## 13. 注意事項・既知の制約

### タイムゾーン問題（解決済み）

**問題**: Firestore から取得したスケジュールの `startDate` は JST (タイムゾーン情報なし) の ISO8601文字列だが、"Z" を付加してUTCとして解釈すると +9時間ずれて日付が翌日になる。

**解決策**: `WidgetData.swift` の `startDateParsed` で、タイムゾーン情報がない場合は `DateFormatter` を使って JST として解釈（`"Z"` を付けない）。

### タグ色のレースコンディション（解決済み）

**問題**: アプリ起動時、`allSchedulesProvider` が `tagsProvider` より先に更新されると、タグ色が空のまま（`colorHex: null`）でキャッシュされる。

**解決策**: `CalendarController` が `tagsProvider` の変更も監視し、タグ更新時にスケジュールを再キャッシュ。

```dart
// calendar_provider.dart
_ref.listen<List<TagItem>>(tagsProvider, (_, __) {
  final schedules = _ref.read(allSchedulesProvider);
  schedules.whenData((s) => _cacheSchedules(s));
});
```

### Xcode でのビルド（Runner.xcworkspace を使用）

**重要**: iOS のビルドは必ず `Runner.xcworkspace` を開くこと。`Runner.xcodeproj` を直接開くと Pod フレームワークが DerivedData に存在せずビルドエラーになる。

```bash
open ios/Runner.xcworkspace  # ✅ 正しい
open ios/Runner.xcodeproj   # ❌ ビルドエラーになる
```

### WidgetKit の制限

- ウィジェットはアプリとは別プロセスで動作するため、Firestore SDK を直接使用不可
- GeometryReader 非対応のため画面サイズを動的取得できない（固定サイズ値を使用）
- ネットワーク通信不可（共有ストレージからの読み取りのみ）
- アニメーション制限あり

### Android の制限

- RemoteViews を使用するため、SwiftUI のような柔軟なレイアウトは不可
- 週数が変わるとレイアウトXMLを切り替える必要がある（`w0`〜`w5`）
- Kotlin から直接 Firestore にアクセス不可（Flutter 経由でデータ取得）
