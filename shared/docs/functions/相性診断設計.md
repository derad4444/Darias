## 機能概要

フレンドと自分の BIG5 性格スコアを比較し、**4つのカテゴリ**（友情・恋愛・仕事・信頼）ごとに相性診断を提供する。

### カテゴリ一覧

| カテゴリキー | 表示名 | 診断観点 |
|------------|--------|---------|
| `friendship` | 友情 | 友人としての日常的な付き合い方・笑いのセンス・困った時に頼れるか |
| `romance` | 恋愛 | 感情的な共鳴・ドキドキ感と安心感のバランス・愛情表現のスタイル |
| `work` | 仕事 | 役割分担・アイデア出し・締め切り意識・お互いの強みの活かし方 |
| `trust` | 信頼 | 誠実さ・約束を守る姿勢・正直に話せるか・長期的な信頼関係 |

### 各診断の出力

- **スコア**: 0〜100% の数値（BIG5から決定論的に算出）
- **キャラクター会話**: 自分と相手のキャラクターが4〜5ターン語り合う会話（AI生成）
- **コメント**: 相性の現状を表す30文字以内の短文（AI生成）
- **アドバイス**: 相性を活かすための60文字以内の具体的提案（AI生成）

## フロー設計

### ユーザーフロー

```
フレンド詳細画面
    ↓
「相性診断」ボタン
    ↓
CompatibilityScreen（カテゴリ選択画面）
    ├── 既存スコア表示（全4カテゴリのスコアバー）
    │   ※ スコアは双方のBIG5から即時算出（API不要）
    └── 各カテゴリカード
        ├── 未解放: 「リワード広告で診断」ボタン
        └── 解放済み: 診断結果サマリー → タップで詳細画面へ
            ↓（未解放の場合）
        リワード広告を視聴
            ↓ 視聴完了
        Cloud Function `diagnoseCompatibility` を呼び出し
            ↓（約2〜5秒）
        CompatibilityCategoryScreen（カテゴリ詳細画面）
        └── アニメーション付きで会話 → スコア → コメント → アドバイスを表示
```

### スコア先行表示の仕組み

スコアは Cloud Function を呼び出す前にクライアント側で計算できる（決定論的アルゴリズム）。
ただし現在の実装では Cloud Function が返す `scores` マップをそのまま使用している。

将来の最適化: クライアント側でスコアのみ先行計算し、API呼び出し後に会話・コメントを追加表示する形にすることでUXを向上できる。

## スコア算出アルゴリズム

### 類似度関数

```
calcSimilarity(a, b) = max(0, 100 - |a - b| × 20)
```

BIG5スコアは1〜5の整数。差分0なら100%、差分1なら80%、差分5なら0%。

### カテゴリ別スコア計算式

#### 友情 (friendship)
```
friendship = round(
    calcSimilarity(my.E, friend.E) × 0.40
  + (my.A + friend.A) / 2 × 20 × 0.40
  + (100 - |my.N - friend.N| × 15) × 0.20
)
```
- **外向性の類似度** (40%): 社交スタイルが合うか
- **協調性の平均** (40%): お互いに気遣えるか
- **神経症傾向の差** (20%): 精神的な安定感の差

#### 恋愛 (romance)
```
romanceBase = round(
    calcSimilarity(my.N, friend.N) × 0.40
  + (my.A + friend.A) / 2 × 20 × 0.35
  + calcSimilarity(my.O, friend.O) × 0.25
)
romance = clamp(romanceBase, 20, 82)
```
- **神経症傾向の類似度** (40%): 感情の揺れが似ているか
- **協調性の平均** (35%): お互いを大切にする姿勢
- **開放性の類似度** (25%): 趣味や価値観の共有
- **上限82%/下限20%**: 恋愛相性に現実感を持たせるクランプ

#### 仕事 (work)
```
work = round(
    (my.C + friend.C) / 2 × 20 × 0.50
  + calcSimilarity(my.O, friend.O) × 0.50
)
```
- **誠実性の平均** (50%): 責任感・締め切り意識
- **開放性の類似度** (50%): アイデアの発想スタイル

#### 信頼 (trust)
```
trust = round(
    (my.A + friend.A) / 2 × 20 × 0.40
  + (100 - (my.N + friend.N) / 2 × 15) × 0.35
  + (my.C + friend.C) / 2 × 20 × 0.25
)
```
- **協調性の平均** (40%): 誠実さ・気遣い
- **神経症傾向の低さ** (35%): 安定した関係
- **誠実性の平均** (25%): 約束を守る姿勢

#### 総合 (overall)
```
overall = round((friendship + romance + work + trust) / 4)
```

全スコアは `clamp(0, 100)` で0〜100の範囲に収める（romance以外）。

### 変数の対応
| 変数 | BIG5特性 |
|------|---------|
| O | 開放性 (Openness) |
| C | 誠実性 (Conscientiousness) |
| E | 外向性 (Extraversion) |
| A | 協調性 (Agreeableness) |
| N | 神経症傾向 (Neuroticism) |

## Cloud Function 設計

### 関数仕様

| 項目 | 値 |
|-----|-----|
| 関数名 | `diagnoseCompatibility` |
| ソース | `const/diagnoseCompatibility.js` |
| API バージョン | v2 (`firebase-functions/v2/https`) |
| リージョン | `asia-northeast1` |
| メモリ | `512MiB` |
| タイムアウト | `90秒` |
| モデル | `gpt-4.1-mini` |
| temperature | `0.75` |
| max_tokens | `600` |

### 処理フロー

```
1. 入力検証
   - userId, friendId, category の存在確認
   - category が "friendship"/"romance"/"work"/"trust" のいずれかであることを確認

2. BIG5情報取得（並列）
   - fetchCharacterInfo(userId)
   - fetchCharacterInfo(friendId)
   → users/{uid}/characters/{charId}/details/current の confirmedBig5Scores を取得

3. スコア計算（同期・決定論的）
   - calcGenreScores(myInfo, friendInfo)
   - 全4カテゴリ + overall を算出

4. BIG5キー生成（キャッシュキー）
   - 各ユーザーの BIG5 指紋を生成: "o{O}c{C}e{E}a{A}n{N}"
   - 2つの指紋をアルファベット順でソートして "|" で連結
   - needsFlip = ソート後に順序が変わったかどうか（会話の isMyCharacter 反転に使用）

5. カテゴリ別キャッシュ確認
   - キャッシュキー: "{big5Key}_{category}"
   - Firestore: compatibilityCache/{cacheKey}
   ├── キャッシュヒット:
   │   - conversation の isMyCharacter を needsFlip で反転
   │   - saveToUserDoc() でユーザードキュメントに保存
   │   - 結果を返す
   └── キャッシュミス: → 6へ

6. AI 生成（キャッシュミス時）
   - buildSpeechStyle() で双方の口調・一人称をプロンプト化
   - JSON 形式で {conversation, comment, advice} を生成
   - フォールバック: GPT 失敗時はデフォルト会話を使用

7. キャッシュ保存
   - compatibilityCache/{cacheKey} に保存

8. ユーザードキュメント保存
   - users/{userId}/compatibilityResults/{friendId} に merge 保存
   - FieldValue.arrayUnion でカテゴリを unlockedCategories に追加

9. 結果を返す
   - {comment, advice, conversation, big5Key, scores}
```

### プロンプト設計

#### キャラクター口調の生成 (`buildSpeechStyle`)

BIG5スコアと性別から自動的に口調・一人称を決定する。

```
性別: 女性 → 一人称「私」
      男性: E≥4 → 一人称「俺」 / E≤2 → 「僕」 / それ以外 → 「僕」

口調パターン（女性）:
  E≥4: 明るく社交的、「〜だよ」「〜ね」「〜かな」
  E≤2: おっとり、「〜かな」「〜ね」「〜だと思う」
  その他: 自然体、バランスよく

追加修飾:
  A≥4 → 「相手を気遣う優しい表現を好む」
  N≥4 → 「少し心配性で感情豊かな面がある」
  O≥4 → 「好奇心旺盛で新しいことに興味を示す」
  C≥4 → 「物事をしっかり考えてから話す」
```

#### システムプロンプト（概要）

- カテゴリ（友情/恋愛/仕事/信頼）特有のトピックに集中
- 会話4〜5ターン、交互
- キャラクター名は「自分」「相手」（固有名詞使わない）
- JSON形式: `{conversation: [{isMyCharacter, text}], comment, advice}`

## Firestore スキーマ

### `users/{userId}/compatibilityResults/{friendId}`

診断実行ユーザー視点の結果を保存する。

```json
{
  "scores": {
    "friendship": 78,
    "romance": 65,
    "work": 82,
    "trust": 71,
    "overall": 74
  },
  "unlockedCategories": ["friendship", "work"],
  "friendship": {
    "comment": "自然体で一緒にいられる良い関係",
    "advice": "定期的に気軽な会話を続けると絆が深まる",
    "conversation": [
      {"isMyCharacter": true, "text": "友達としての相性は78%だって！"},
      {"isMyCharacter": false, "text": "そうなんだ、なんか嬉しいね"},
      ...
    ],
    "big5Key": "o2c3e4a3n2|o3c4e3a4n2",
    "createdAt": Timestamp
  },
  "work": {
    "comment": "仕事でのチームワークは抜群",
    ...
  }
}
```

**書き込みルール:**
- `set({merge: true})` で追記的に保存
- 新しいカテゴリを診断するたびにそのカテゴリフィールドが追加される
- `scores` は最新の診断で上書き（BIG5スコア変化時も対応）
- `unlockedCategories` は `FieldValue.arrayUnion` で追記（削除なし）

### `compatibilityCache/{cacheKey}`

AIが生成したカテゴリ別会話のキャッシュ。

```
ドキュメントID例: "o2c3e4a3n2|o3c4e3a4n2_friendship"
```

```json
{
  "comment": "自然体で一緒にいられる良い関係",
  "advice": "定期的に気軽な会話を続けると絆が深まる",
  "conversation": [
    {"isMyCharacter": true, "text": "友達としての相性は78%だって！"},
    {"isMyCharacter": false, "text": "そうなんだ、なんか嬉しいね"},
    ...
  ],
  "big5Key": "o2c3e4a3n2|o3c4e3a4n2",
  "createdAt": Timestamp
}
```

**キャッシュの対称性:**
- BIG5指紋はソートされているため、A↔B と B↔A は同じキャッシュドキュメントを参照する
- 再利用時は `needsFlip` フラグで `isMyCharacter` を反転することで、視点を正しく切り替える

### Firebase Functions レスポンスの型変換

Firebase Functions は `Map<Object?, Object?>` を返すが Dart は `Map<String, dynamic>` を期待するため、JSON round-trip で変換する必要がある。

```dart
final data = jsonDecode(jsonEncode(result.data)) as Map<String, dynamic>;
return CategoryDiagnosis.fromFunctionResult(data, category);
```

## キャッシュ戦略

### キャッシュの目的

1. **コスト削減**: 同じBIG5組み合わせへの重複API呼び出しを防ぐ
2. **レスポンス向上**: キャッシュヒット時は Firestore 読み取りのみ（< 1秒）

### キャッシュキーの設計

```
big5Key = sort([fingerprint(userA), fingerprint(userB)]).join("|")
cacheKey = big5Key + "_" + category
```

```
fingerprint(user) = "o{O}c{C}e{E}a{A}n{N}"
例: "o3c4e2a5n1"
```

キーをソートすることで `A診断B` と `B診断A` が同じキャッシュを参照する。

### isMyCharacter の反転

キャッシュは「最初に診断したユーザーの視点」で保存されている。後から参照する場合（自分が B の場合）、`isMyCharacter` を反転する必要がある。

```javascript
const needsFlip = sorted[0] !== myFp; // myFpがソート後に後ろになった場合
const conversation = cached.conversation.map(msg => ({
  ...msg,
  isMyCharacter: needsFlip ? !msg.isMyCharacter : msg.isMyCharacter,
}));
```

### キャッシュヒット率の考察

BIG5は各特性1〜5の整数なので1ユーザーのフィンガープリントは `5^5 = 3,125` 通り。
2ユーザーのペアは最大 `3,125^2 / 2 ≒ 488万` 通り（対称性考慮）× 4カテゴリ。

6人会議と異なり同じ性格タイプ同士のペアは少なく、キャッシュヒット率は低い（初期: < 5%）。
ただしコストが安いためヒット率が低くても問題ない。
